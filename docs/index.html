Resumo executivo do projeto Metaflake
O que é o Metaflake▾

Metaflake é uma ferramenta open-source de documentação de pipelines ETL para Data Warehouses, distribuída como um único arquivo HTML autocontido — zero instalação, zero dependências, zero build.

A ideia é simples: qualquer engenheiro de dados abre o arquivo no navegador e começa a documentar seu pipeline (origens, dimensões, fatos, regras de mapeamento, procedimentos, agendamentos). O resultado pode ser salvo como um novo HTML e compartilhado com a equipe.

A ferramenta serve como trampolim de marketing para o Metaflake Pro, produto comercial da Data Praxes com lineage visual, catálogo corporativo, versionamento Git e export DBT.

Princípios de Design▾
  1. Zero fricção: abrir o arquivo e usar. Sem cadastro, sem servidor, sem instalação.
  2. Visual premium: a ferramenta open source deve parecer um produto pago — estética dark glassmorphism, animações sutis, atenção a tipografia.
  3. Portabilidade total: o documento gerado é o próprio app — pode ser enviado por email, commitado no Git, aberto offline.
  4. Persistência local: localStorage evita perda de trabalho entre sessões.
  5. Upgrade natural: expor features do Pro em pontos estratégicos (sidebar CTA, footer, hints), sem bloquear o uso core.
  6. Documentação embutida: o template vem com dados-exemplo realistas para o usuário entender o uso sem ler manual.

PRD · Product Requirements Document
O que estamos construindo, para quem, e por quê
1.1 Problema▾

Equipes de engenharia de dados documentam seus pipelines ETL de forma dispersa e ineficiente:

  • Confluence/Notion/Wiki: texto livre que envelhece rápido, fica desatualizado em ciclos de sprint, difícil de versionar.
  • Comentários no código (SQL/Python): contextuais mas invisíveis para stakeholders não-técnicos (BI, produto, auditoria).
  • Excel/Word soltos em repositórios: difíceis de manter, sem integridade, sem visualização gráfica.
  • Ferramentas corporativas (Collibra, Alation, Monte Carlo): caras, exigem implantação longa, overkill para times pequenos/médios.
⚠️
Dor central
Não existe uma forma leve, gratuita e bonita de documentar um pipeline ETL que seja simultaneamente útil para o engenheiro (rápido de preencher, versionável) e para o stakeholder (navegável, navegável, apresenta linhagem).
1.2 Solução▾

Um single-page HTML app que oferece um formulário estruturado (Informações Gerais, Domínios, Demandas, Mapeamentos, Arquitetura, Procedimentos, Histórico) e renderiza o resultado como uma página navegável com sidebar, seções colapsáveis e barra de progresso.

O usuário pode:

  • Preencher via modo Editar (formulário com validação visual).
  • Visualizar via modo Visualizar (página limpa para leitura).
  • Salvar Arquivo — baixa um novo HTML com os dados embutidos no seed-data.
  • Recarregar o HTML e continuar de onde parou (via localStorage).
✅
Diferencial
A documentação é o app. Não existe separação entre "editor" e "documento" — o mesmo HTML serve para criar e para ler. Isso elimina o problema de sincronização entre wiki e realidade.

Como o Metaflake é construído por dentro
2.1 Arquitetura▾

O Metaflake segue um padrão single-file SPA com três camadas embutidas no mesmo HTML:

┌────────────────────────────────────────────────────────┐
│  metaflake.html                                        │
│                                                        │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │  <style>     │  │  seed-data   │  │  <script>    │  │
│  │  CSS         │  │  JSON        │  │  JS          │  │
│  │  (~700 lin.) │  │  (~10 KB)    │  │  (~2700 lin.)│  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
│                                                        │
│  ┌────────────────────────────────────────────────┐    │
│  │  <body> — HTML da UI (sidebar, toolbar, main)  │    │
│  └────────────────────────────────────────────────┘    │
└────────────────────────────────────────────────────────┘

Fluxo de dados

  1. No load: defaultData = JSON.parse(seed-data.textContent)
  2. Tenta carregar de localStorage (se existir, sobrescreve defaultData).
  3. populateView() preenche os campos do modo Visualizar.
  4. loadIntoForm() preenche os campos do modo Editar.
  5. Em cada oninput, updateView() re-renderiza o view + saveData() escreve no localStorage + updateProgressBar() atualiza dashboard.
  6. "Salvar Arquivo" serializa buildFullData() de volta para o seed-data e baixa o HTML completo.
ℹ️
Por que single file?
Zero fricção de distribuição. Sem npm, sem build, sem CDN. O usuário pode baixar, commitar no Git, enviar por Slack — funciona em qualquer lugar que tenha um navegador.

2.4 Schema do seed-data▾

O JSON embutido em <script type="application/json" id="seed-data"> é a fonte de verdade do documento. Estrutura:

{
  "projeto_datastage_topo": string,        // Nome do projeto
  "assunto_topo": string,                  // Assunto/tema
  "dominios_topo": string,                 // Domínios separados por \n
  "ambiente_producao": string,
  "ambiente_homologacao": string,
  "ambiente_desenvolvimento": string,
  "equipe_etl_responsavel": string,
  "lider_responsavel": string,
  "descricao_assunto": string,             // Markdown leve
  "tables": {
    "table-palavras_chaves": [{ assunto, palavras_chaves }],
    "table-contatos":      [{ nome, papel, email, telefone }],
    "table-historico":     [{ versao, data, autor, alteracao }]
  },
  "demandas":          [{ dominio, descricao }],          // por domínio
  "relacionamentos":   [{ fato, tipo_carga, defasagem, dimensoes: [{tabela, tipo_tabela}] }],
  "regrasMapeamento":  [{ job, dominio, tipo_job, regras: [{tabela_origem, campo_origem, tabela_destino, campo_destino, regra, obs}] }],
  "narrativaPassos":   [{ titulo, subitens: [string] }],
  "detalhesTecnicos":  [{ titulo, texto }],
  "execucaoItens":     [{ titulo, seq_localizacao, seq_descricao, precondicao, poscondicao, parametros: [{nome, valor}] }],
  "reprocessamentoItens": [{ titulo, texto }],
  "erroItens":         [{ titulo, texto }],
  "validacaoItens":    [{ titulo, texto }],
  "agendamentoItens":  [{ ambiente, periodicidade, evento, horario }],
  "cadastroItens":     [{ cd_obj, cd_prc, dominio, ds_obj }],
  "arquiteturaItens":  [{ titulo, dominio, texto }],
  "fluxoDadosItens":   [{ titulo, dominio, texto }],
  "tabelasCadastradas": [string]                        // auto-coletado dos relacionamentos/regras
}
ℹ️
Convenção importante
tabelasCadastradas é auto-gerado a partir dos relacionamentos e regrasMapeamento — não deve ser editado manualmente. A função buildFullData() consolida-o antes de salvar.

Variáveis CSS (theme tokens)

Toda a identidade visual é controlada por CSS variables em :root (tema claro) e [data-theme="dark"] (tema escuro — default).

:root {
    --bg-body:        #f5f7fb;
    --bg-card:        rgba(255,255,255,0.85);
    --accent-primary: #7c3aed;       /* violet */
    --accent-secondary: #0ea5e9;     /* sky */
    --gradient-brand: linear-gradient(135deg, #7c3aed 0%, #00d4fc 100%);
    --radius-lg: 14px;
    --glass-blur: 12px;
    /* ... */
}
[data-theme="dark"] {
    --bg-body:        #0b0e14;
    --accent-primary: #a78bfa;
    --accent-secondary: #00d4fc;
    /* ... */
}

Componentes principais

  • .toolbar — topbar com logo + busca + ações
  • .sidebar + .sidebar-brand + .sidebar-cta-pro + .sidebar-user
  • .doc-header — header com greeting + título gradient
  • .stats-grid + .stat-card (com variantes .stat-violet, .stat-cyan, .stat-emerald, .stat-amber)
  • .progress-ring-widget — widget com SVG circular animado

Glassmorphism

Aplicado em toolbar, sidebar, doc-header, sections, stat cards, widgets via:

background: var(--bg-card);  /* rgba com transparência */

SKILLS · Competências Necessárias
O que o time precisa dominar para evoluir o produto
3.1 Frontend▾
  • HTML5 semântico — estrutura correta de <section>, <nav>, <main>, <article> para acessibilidade.
  • CSS modern — backdrop-filter, grid, flexbox, custom properties, clamp(), media queries.
  • JavaScript vanilla ES2020+ — manipulação de DOM, fetch (não usado ainda, mas previsto), async/await, localStorage, File System Access API.
  • SVG inline — desenho de gráficos (star schema, lineage, progress ring) sem libs.
  • Acessibilidade (WCAG AA) — contraste, foco visível, navegação por teclado, ARIA onde necessário.
  • Performance web — Critical CSS inline, evitar reflow, debounce em inputs grandes.
3.2 Design / UX▾
  • Design system thinking — definir tokens (cores, raios, sombras, espaçamentos) e usá-los consistentemente.
  • Glassmorphism — equilibrar blur, transparência e saturação sem prejudicar legibilidade.
  • Hierarquia visual — tipografia escalonada (28px título, 16px sub, 13px corpo, 11px label).
  • Microinterações — hover lift, transições suaves (cubic-bezier), feedback visual imediato.
  • Dark mode — escolher cores de accent que funcionem em ambos os temas (roxo/ciano).
  • Mobile-first — breakpoints em 1024px e 768px, layout reflow para sidebar virar dropdown.
  • Ferramentas: Figma para mockups, SVG para ícones, Google Fonts (Inter).
3.3 Domínio ETL / DW▾

Para que o produto faça sentido, é preciso entender o domínio. Recomendado:

  • Modelagem dimensional Kimball — star schema, fato vs dimensão, grain, SCD tipos 1/2/3, role-playing dimensions, conformed dimensions.
  • ETL clássico — staging → integration → mart, surrogate keys, lookup, erro handling com sk = -1 (placeholders).
  • Padrões de carga — full vs incremental, D-1/D-0, janela de reprocessamento, idempotência.
  • Qualidade de dados — validações de contagem, soma, integridade referencial, rejeição e quarentena.
  • Ferramentas de referência — SAP SD/SD (origem comum), Airflow, dbt, DataStage, SSIS, Informatica.
  • catálogo/lineage — entendimento do problema que Collibra/Alation resolvem (para diferenciar o Pro).

Leitura recomendada

MEMÓRIA · Contexto & Decisões
Por que fizemos o que fizemos — para não esquecer
4.1 Contexto do Projeto▾

O doc-dw nasceu como um template interno de documentação ETL usado pela Data Praxes em projetos com clientes (ex.: Sicoob). Em julho de 2026, decidiu-se elevá-lo a produto open source com branding próprio (Metaflake) para:

  • Gerar autoridade técnica no mercado de dados brasileiro.
  • Criar um topo de funil qualificado para o produto comercial (Metaflake Pro).
  • Estabelecer um padrão de qualidade visual que diferencia a Data Praxes de concorrentes.
🎯
Estratégia
Open source como marketing — a ferramenta gratuita deve ser tão boa que o usuário pensa "se a versão grátis é assim, imagine a paga".
4.2 ADRs · Architecture Decision Records▾
ADR-001 Single HTML file como formato de distribuição Aceito
Data
2026-07-12
Status
Aceito
Decisor
Squad Data Praxes

Contexto: Precisávamos escolher entre (a) single HTML file, (b) app React/Vue com build, (c) app Electron, (d) SaaS web.

Decisão: Single HTML file autocontido.

Consequências:

  • ✅ Zero fricção de adoção — abrir e usar.
  • ✅ Portabilidade total (email, Git, pendrive).
  • ✅ Funciona offline em ambientes corporativos restritos.
  • ❌ Tamanho do arquivo cresce com anexos base64.
  • ❌ Manutenção mais trabalhosa (sem componentização nativa).

Lição 1 — git rm vs git rm --cached
Durante a configuração do repositório, executamos git rm -r backup sicoob sem --cached, o que deletou os arquivos do disco além de removê-los do Git. Felizmente o histórico preserva os blobs em commits antigos — recuperamos com git checkout 0fd4f21 -- caminho e git restore --staged para mantê-los fora do versionamento. Aprendido: sempre --cached quando quiser manter no disco, e sempre --dry-run antes de comandos destrutivos.
⚠️
Lição 2 — Mensagem de commit deve refletir a ação
O commit "Remove backup, sicoob e doc-dw.html do versionamento" sugeria que os arquivos permaneceriam no disco, mas a ação foi "remover do versionamento E do disco". O Git não valida mensagem vs ação. Aprendido: revisar com git status antes de commitar comandos rm.
💡
Lição 3 — Variáveis CSS globais >override seletor a seletor
Redesenhar um app de 3000 linhas mudando apenas as variáveis CSS em :root e [data-theme="dark"] transforma 90% do visual instantaneamente. Componentes específicos recebem override só onde necessário. Aprendido: sempre estruturar design system via tokens primeiro.
💡
Lição 4 — Dashboard com dados vivos > dados estáticos
Os stat cards e o progress ring só fizeram sentido quando conectados via JS aos dados reais (updateDashboardStats). Estáticos pareciam mockup; dinâmicos viram feature. Aprendido: qualquer widget visível deve refletir estado real do app.
